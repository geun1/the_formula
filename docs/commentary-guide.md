# 해외 테크 블로그 — 상세 리뷰(논평) 작성 스킬

> 소유: 민성(수집) · 적용: enrich(`src/lib/cardnews.ts`, source `kind="techblog"` 분기) · 근거: [sourcing-policy.md](./sourcing-policy.md)
> 계보: `paper-review-service`의 **blog-review** 루브릭(트레이드오프·수치 검증·마케팅 vs 실제 기여·재현성)을 단일-패스 enrich 용으로 적응.

## 왜 "요약"이 아니라 "리뷰"인가

테크 블로그는 *"왜 이렇게 설계했나 / 실제로 쓸 만한가"* 가 핵심입니다. 단순 번역·요약이 아니라 **비판적 리뷰**여야 ① 저작권상 안전(재서술·논평이 주 → [sourcing-policy.md](./sourcing-policy.md) 충족, 변형적 성격↑)하고 ② 한국 실무자에게 진짜 가치(이 글을 믿고 도입해도 되나)를 줍니다.

## 산출물 (CardNews) — body 구조

`summary`(2~3문장 티저) / `keywords`(3~5) / `category` / `body`(마크다운). **body 만 상세 리뷰 구조**:

```markdown
## 한눈에 (TL;DR)
무엇을 발표/주장하나 · 왜 중요한가 · 핵심 수치. 5~7문장으로 압축.

## 핵심 주장
이 글이 내세우는 주장/기여 2~4개를 불릿으로. 제품·엔지니어링 주장 형태
(예: "성능 N% 개선", "X 기능 최초 지원", "Y 비용 Z% 절감").

## 어떻게 풀었나
접근·설계의 핵심을 **에디터 자신의 한국어로 재서술**. 원문 문단 통째 번역 금지.
꼭 필요한 핵심 문장만 '> ' 인용으로 1~3개 짧게. 수치 비교는 표/```chart```(원문 수치만).

## 뜯어보기 (비판적 검토)   ← 이 섹션이 '리뷰'의 핵심
- **트레이드오프**: 이 설계로 무엇을 포기했나
- **수치의 측정 조건**: 주장하는 성능·개선의 비교 대상/하드웨어/데이터가 공정한가
- **마케팅 vs 실제 기여**: 포지셔닝과 진짜 기술 기여의 경계
- **재현·검증**: 코드·벤치마크·데이터가 공개됐나, 제3자가 확인 가능한가

## 실무 적용
한국 개발·AX 실무자가 도입한다면: 걸림돌은 무엇이고, 다음 행동(또는 더 읽을 자료)은?

## 한 줄 평
핵심 주장 한 줄 요약 + 가장 미심쩍거나 과장으로 보이는 지점 1개.

> 전문은 원문 출처에서 확인하세요.
```

## 원칙

1. **재서술 우선**: body 대부분은 번역이 아니라 에디터의 요약·해석·비평. 원문이 주(主)가 되면 안 됨.
2. **발췌 인용은 짧게**: 핵심 문장만 1~3개 '>' 인용. 전체 분량 복제 금지.
3. **비판은 근거 기반**: '뜯어보기'의 의심은 본문 내용에 근거. 없는 사실·수치 창작 금지(발췌만 있으면 있는 내용만).
4. **출처로 유도**: 끝에 원문 한 줄. 실제 링크는 기사 페이지가 `원문 보기 · {sourceName} →`로 렌더하므로 URL 미주입.
5. **해요체**, 소제목 구조화.

## 적용 메커니즘

- 소스 `kind="techblog"` → enrich 가 이 리뷰 프롬프트 사용(`getSourceKind(sourceName)`).
- 그 외 소스(newsletter/blog/geeknews)는 기존 동작(충실 재구성) 유지 — 변경 격리.
- 리뷰 모드를 다른 해외 소스로 넓히려면 해당 소스 `kind` 를 `techblog` 로.
- 실제 예시: [examples/commentary-netflix-causal-inference.md](./examples/commentary-netflix-causal-inference.md)
