# 해외 테크 블로그 — 논평(Commentary) 작성 스킬

> 소유: 민성(수집) · 적용: enrich(`src/lib/cardnews.ts`, source `kind="techblog"` 분기) · 근거: [sourcing-policy.md](./sourcing-policy.md)
> 이 문서는 `techblog` 소스를 발행할 때 따르는 **편집 규칙(스킬 정의)** 입니다. enrich 프롬프트가 이 규칙을 구현합니다.

## 왜 논평인가

해외 테크 블로그(Netflix·Pinterest·Meta 등)는 회사 저작물이며 전문 번역·재발행 라이선스가 없습니다([sourcing-policy.md](./sourcing-policy.md)). 따라서 **전문 번역이 아니라 "발췌 + 논평 요약"** 으로 발행합니다. 이는 저작권법 제28조(공표 저작물의 인용)에 부합하는 방향이자, "원문을 대체하지 않고 원문으로 보내는" 큐레이션 포지션입니다.

## 산출물 형태 (CardNews)

`summary` / `keywords` / `body`(마크다운) / `category` — 스키마는 기존 `cardNewsTextSchema` 그대로. **body 의 성격만 다름**(전문 번역 → 논평).

### body 구조

```markdown
## 한눈에
3~4줄. 이 글이 무엇을 했고 왜 읽을 만한지.

## 무엇을 다루나
글의 내용을 **에디터 자신의 한국어로 재서술**(요약). 원문 문단 통째 번역 금지.
꼭 필요한 핵심 문장만 1~3개, 짧게 발췌·번역해 '>' 인용으로 제시.
수치 비교가 있으면 표 또는 ```chart``` 활용(원문에 있는 수치만).

## 왜 중요한가 / 실무 시사점
한국 개발·AX 실무자 관점의 해석. "우리 일에 어떻게 연결되나".

## 한계 · 생각할 점
비판적 시각·적용 시 주의점(있으면).

> 전문은 원문 출처에서 확인하세요.
```

## 원칙

1. **재서술 우선**: 본문의 대부분은 원문 번역이 아니라 에디터의 요약·해석. 원문이 주(主)가 되면 안 됨.
2. **발췌 인용은 짧게**: 핵심 문장만 1~3개, '>' 인용. 원문 전체 분량을 복제하지 않음.
3. **창작 금지**: 원문에 없는 수치·주장 지어내지 않음. 발췌만 있으면 있는 내용만.
4. **출처로 유도**: 본문 끝에 원문으로 보내는 한 줄. 실제 링크는 기사 페이지가 `원문 보기 · {sourceName} →` 로 렌더하므로 본문엔 URL 미주입.
5. **해요체**, 소제목 구조화, 한국 실무자 가독성.

## 적용 메커니즘

- 소스의 `kind="techblog"` 이면 enrich 가 이 논평 프롬프트를 사용(`getSourceKind(sourceName)`).
- 그 외 소스(newsletter/blog/geeknews)는 기존 동작(충실 재구성) 유지 — 변경 격리.
- 논평 모드를 다른 해외 소스로 넓히려면 해당 소스의 `kind` 를 `techblog` 로 바꾸면 됨.
